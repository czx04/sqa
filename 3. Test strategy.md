# 3. TEST STRATEGY

Chiến lược kiểm thử mô tả cách tiếp cận đối với hệ thống **DATN** (backend NestJS + web Next.js + Prisma/DB + tích hợp VNPay, GHN, Recurly, S3). Tham chiếu `FSOFT.GUIDELINE_TEST PLAN.pdf` cho nguyên tắc chung; phần dưới áp dụng cụ thể cho đồ án.

**Khi không có tài liệu yêu cầu độc lập:** phần **kiểm thử chức năng** vẫn dựa trên **kịch bản và luồng nghiệp vụ** nhưng “expected result” được lấy từ **thống nhất nhóm + Swagger + quan sát build**; phần **tĩnh** tăng tỷ trọng đọc mã, route, DTO validation (Zod) thay cho review SRS.

---

## 3.1 Test types

### 3.1.1 Static Testing

Kiểm thử tĩnh bổ sung cho kiểm thử động: rà soát tài liệu yêu cầu, thiết kế API, checklist code (ESLint, TypeScript), có thể dùng công cụ phân tích mã (SonarQube, v.v.) nếu nhóm cấu hình được — **không thực thi** ứng dụng.

| Mục | Nội dung |
|-----|----------|
| **Mục tiêu** | Phát hiện sai lệch so với chuẩn, rủi ro thiết kế trước khi chạy test động; khi không có SRS thì tập trung **mâu thuẫn nội bộ** (API vs DB, web vs API). |
| **Kỹ thuật** | Nếu có SRS/use case: review tài liệu đó. **Nếu không:** đọc module/route/DTO, đối chiếu **Swagger** với hành vi thực tế; ESLint/Prettier trên `DATN_api-main` và `DATN_web_client-main`. |
| **Tiêu chí hoàn thành** | Các mục review theo kế hoạch đã thực hiện; lỗi tài liệu/mã tĩnh đã được xử lý hoặc ghi nhận. |
| **Lưu ý** | Review trùng với Project Plan có thể không lặp lại trong Test Plan chi tiết. |

---

### 3.1.2 Unit Testing

| Mục | Nội dung |
|-----|----------|
| **Mục tiêu** | Đảm bảo từng đơn vị mã (service, util, validator) hoạt động đúng thiết kế. |
| **Kỹ thuật** | **Jest** (`*.spec.ts`) trên API: mock dependency (Prisma, HTTP client), kiểm tra nhánh hợp lệ/không hợp lệ; có thể kết hợp black-box qua input/output và white-box cho nhánh quan trọng. |
| **Tiêu chí hoàn thành** | Các unit test đã lên kế hoạch cho module ưu tiên đã chạy; lỗi mức ưu tiên đã xử lý; mức **coverage** do nhóm quy định (ví dụ ≥ nhóm ngưỡng đồ án) nếu có. |
| **Lưu ý** | Ưu tiên module nghiệp vụ phức tạp (đơn hàng, thanh toán, giảm giá, quyền). |

---

### 3.1.3 Function Testing

#### 3.1.3.1 Function Testing

| Mục | Nội dung |
|-----|----------|
| **Mục tiêu** | Đảm bảo chức năng đúng theo use case: nhập liệu, xử lý, truy vấn, phản hồi API và luồng web. |
| **Kỹ thuật** | Thực thi từng luồng với dữ liệu **hợp lệ / không hợp lệ**; kiểm tra mã lỗi HTTP, thông báo i18n, áp dụng **business rule** (giá, tồn, quyền). Có thể tự động hóa một phần bằng **Jest + supertest** (API) hoặc **Playwright/Cypress** (web) nếu triển khai. |
| **Tiêu chí hoàn thành** | Đủ ca kiểm thử theo mục 2.1 đã thực thi; lỗi chức năng đã xử lý hoặc ghi nhận theo quy trình. |
| **Lưu ý** | Tích hợp bên thứ ba có thể cần mock hoặc sandbox; lỗi mạng ngoài không gán nhầm cho logic nội bộ. |

#### 3.1.3.2 Business Cycle Testing

| Mục | Nội dung |
|-----|----------|
| **Mục tiêu** | Xác minh chuỗi nghiệp vụ theo thời gian và trạng thái (ví dụ: giỏ → đơn → thanh toán → giao hàng → hoàn tất/hủy). |
| **Kỹ thuật** | Lặp luồng end-to-end với nhiều tài khoản; kiểm tra biên thời gian (hết hạn token, khuyến mãi theo ngày); job định kỳ (ví dụ cron dọn refresh token) nếu nằm trong phạm vi. |
| **Tiêu chí hoàn thành** | Các chu kỳ đã chọn chạy ổn định; lỗi đã xử lý. |
| **Lưu ý** | Cần đồng hồ hệ thống và dữ liệu mẫu nhất quán; mô hình nghiệp vụ lấy từ SRS. |

---

### 3.1.4 User Interface Testing

| Mục | Nội dung |
|-----|----------|
| **Mục tiêu** | Điều hướng đúng nghiệp vụ; form, bảng, thông báo lỗi/thành công hiển thị hợp lý; trạng thái focus/tab phù hợp trên web. |
| **Kỹ thuật** | Kiểm thử thủ công theo trang (trang chủ, danh mục, chi tiết sản phẩm, giỏ, thanh toán, tài khoản); kiểm tra responsive cơ bản; đối chiếu với CASL (ẩn/hiện thao tác). |
| **Tiêu chí hoàn thành** | Các màn hình trong phạm vi đã duyệt; không có lỗi chặn luồng chính chưa xử lý. |
| **Lưu ý** | Component thư viện bên thứ ba có thể không kiểm tra hết thuộc tính nội bộ; tập trung hành vi người dùng. |

---

### 3.1.5 Data and Database Integrity Testing

| Mục | Nội dung |
|-----|----------|
| **Mục tiêu** | Truy cập DB qua Prisma đúng: không mất dữ liệu, không cập nhật sai bản ghi, ràng buộc quan hệ đúng. |
| **Kỹ thuật** | Sau giao dịch (đặt hàng, thanh toán, đổi trạng thái), đối chiếu bảng liên quan; thử dữ liệu biên (số âm, trùng SKU, v.v.); có thể dùng Prisma Studio hoặc truy vấn chỉ đọc trên DB test. |
| **Tiêu chí hoàn thành** | Không phát hiện hỏng dữ liệu trong các kịch bản đã test. |
| **Lưu ý** | Dùng bộ dữ liệu nhỏ, có thể reset bằng migrate/seed; tránh sửa trực tiếp production. |

---

### 3.1.6 Performance testing

#### 3.1.6.1 Performance testing (profiling cơ bản)

| Mục | Nội dung |
|-----|----------|
| **Mục tiêu** | Đo thời gian phản hồi một số API và trang quan trọng dưới tải **sử dụng dự kiến** của đồ án (không bắt buộc mô phỏng hàng nghìn user trừ khi SRS yêu cầu). |
| **Kỹ thuật** | Dùng lại kịch bản chức năng; đo thủ công hoặc script (k6, Apache Bench) với vài luồng song song; ghi nhận P95 tham chiếu. |
| **Tiêu chí hoàn thành** | Các giao dịch mục tiêu hoàn thành trong ngưỡng nhóm thống nhất (ví dụ dưới 3–5 giây trên môi trường lab). |
| **Lưu ý** | Chạy khi máy chủ ít nhiễu; DB nên có kích thước gần thực tế hoặc scale tương đương nếu so sánh. |

#### 3.1.6.2 Load Testing

| Mục | Nội dung |
|-----|----------|
| **Mục tiêu** | Đánh giá hành vi khi tăng số request đồng thời vừa phải (ví dụ vài chục luồng). |
| **Kỹ thuật** | Tái sử dụng kịch bản chức năng; tăng số lần lặp hoặc user ảo bằng công cụ load. |
| **Tiêu chí hoàn thành** | Không lỗi hàng loạt; thời gian trong ngưỡng chấp nhận được. |
| **Lưu ý** | Throttler của NestJS có thể trả 429 — phân biệt với lỗi server. |

#### 3.1.6.3 Stress Testing

| Mục | Nội dung |
|-----|----------|
| **Mục tiêu** | Quan sát hệ thống khi tài nguyên căng (CPU/RAM) hoặc tải vượt ngưỡng bình thường; ghi nhận điểm suy giảm. |
| **Kỹ thuật** | Tăng tải đến khi timeout/lỗi; có thể giới hạn RAM máy ảo (nếu an toàn). |
| **Tiêu chí hoàn thành** | Có báo cáo ngưỡng và hành vi (không bắt buộc “không fail” nếu mục tiêu là tìm giới hạn). |
| **Lưu ý** | Chỉ thực hiện trên môi trường test. |

#### 3.1.6.4 Volume Testing

| Mục | Nội dung |
|-----|----------|
| **Mục tiêu** | Kiểm tra với **khối lượng dữ liệu lớn** (nhiều sản phẩm, đơn, log) nếu có kịch bản. |
| **Kỹ thuật** | Seed dữ liệu lớn hoặc import mẫu; chạy truy vấn/tìm kiếm/báo cáo song song. |
| **Tiêu chí hoàn thành** | Ứng dụng không crash; thời gian chấp nhận được theo thỏa thuận. |
| **Lưu ý** | Thời gian “chấp nhận được” cho tải lớn cần định nghĩa trước. |

---

### 3.1.7 Security and Access Control Testing

| Mục | Nội dung |
|-----|----------|
| **Mục tiêu** | **Application-level:** mỗi vai trò chỉ truy cập API/màn hình/dữ liệu được phép. **System-level:** chỉ người được cấp quyền vào hệ thống (JWT hợp lệ, không bypass route). |
| **Kỹ thuật** | Ma trận user (khách, user đăng nhập, admin, seller…); gọi API với token sai/thiếu; thử IDOR (truy cập đơn/sản phẩm của user khác); kiểm tra CORS/Helmet cơ bản. |
| **Tiêu chí hoàn thành** | Mỗi loại actor đã có ca kiểm thử đại diện; không lộ thao tác nhạy cảm cho sai vai trò. |
| **Lưu ý** | Bảo mật hạ tầng mạng có thể do quản trị hệ thống; đồ án tập trung lớp ứng dụng. |

---

### 3.1.8 Regression Testing

| Mục | Nội dung |
|-----|----------|
| **Mục tiêu** | Sau khi sửa lỗi hoặc refactor, phần cũ không bị hỏng. |
| **Kỹ thuật** | Tái sử dụng bộ TC đã có; ưu tiên **smoke** và TC **cao** cho luồng TMĐT; có thể tự động hóa nhóm TC lặp lại nhiều. Chọn ngẫu nhiên một tỷ lệ TC cũ (ví dụ ~80%) cho bản build lớn nếu nhóm áp dụng chiến lược mẫu FSOFT. |
| **Tiêu chí hoàn thành** | TC hồi quy chọn chạy đạt; không phát sinh lỗi nghiêm trọng trên vùng liên quan. |
| **Lưu ý** | Phạm vi hồi quy phụ thuộc mức độ thay đổi (diff module). |

---

### 3.1.9 Interfaces or connections testing

| Mục | Nội dung |
|-----|----------|
| **Mục tiêu** | Tích hợp với **VNPay**, **GHN**, **Recurly**, **AWS S3**, email/Redis (nếu dùng) hoạt động đúng luồng; người dùng không gặp lỗi do cấu hình sai callback/URL. |
| **Kỹ thuật** | Kịch bản sandbox: thanh toán thành công/thất bại, webhook/callback, tạo đơn vận chuyển, upload ảnh; xử lý lỗi mạng và mã lỗi từ nhà cung cấp. |
| **Tiêu chí hoàn thành** | Các luồng tích hợp trong phạm vi đã test pass; biên được ghi nhận. |
| **Lưu ý** | Phụ thuộc khóa API và chính sách sandbox từng bên. |

## 3.2 Test stages

Các giai đoạn áp dụng: **Unit → Integration → System → Acceptance** (có thể lặp theo sprint/build).

### Bảng tiêu chí giai đoạn (rút gọn cho đồ án)

| Test level | Nhiệm vụ chính | Tiêu chí vào | Tiêu chí tạm dừng | Tiêu chí tiếp tục | Tiêu chí ra |
|------------|----------------|-------------|------------------|------------------|------------|
| **Unit** | Kiểm đơn vị mã, coverage mục tiêu | Yêu cầu/module rõ; code có thể test được | Build lỗi nặng; đổi yêu cầu lớn; môi trường hỏng | Khắc phục nguyên nhân tạm dừng | Unit test đã chạy; lỗi ưu tiên đã xử lý |
| **Integration** | API + DB + mock/tích hợp dịch vụ ngoài | Unit xong; môi trường và dữ liệu test sẵn sàng | Lỗi chặn; thiếu tài nguyên | Ổn định build/môi trường | Không lỗi critical mở; luồng tích hợp chính pass |
| **System** | E2E web + API theo yêu cầu | Integration xong; TC/system test sẵn | Lỗi chặn; đổi phạm vi lớn | Khắc phục | ≥ 95% TC hệ thống pass (điều chỉnh theo quy định nhóm); TC ưu tiên cao pass hết |
| **Acceptance** | Nghiệm thu theo tiêu chí đồ án/giảng viên | System test baseline; stakeholder đồng ý | Rủi ro chấp nhận được | Sau khi điều kiện đạt | Đủ tiêu chí mục 2.2; ký nhận/bàn giao |

### Ma trận loại kiểm thử theo giai đoạn

| Type of Tests | Unit | Integration | System | Acceptance |
|---------------|------|-------------|--------|------------|
| Static / Review | Có thể | Có thể | Có thể | Có thể |
| Function | Có | Có | Có | Có |
| User Interface | — | Một phần | Có | Có |
| Performance (profiling, load nhẹ) | — | Một phần | Có | Theo yêu cầu |
| Load / Stress / Volume | — | — | Tùy chọn | Tùy chọn |
| Security | — | Có | Có | Có |
| Data integrity | — | Có | Có | Có |

## 3.3 Automation test strategy

**Hướng án:**

- **API:** Jest + supertest cho các endpoint ổn định; mock Prisma/HTTP khi cần.
- **Web:** Playwright hoặc Cypress cho luồng mua hàng và đăng nhập lặp lại nhiều (nếu nhóm triển khai).
- **Mục tiêu tỷ lệ tự động hóa:** Do nhóm đặt (ví dụ 30–50% TC lặp lại cho API), không bắt buộc toàn bộ.

**Chính sách chọn TC tự động hóa (theo tinh thần Guideline FSOFT):**

| Ưu tiên tự động | Hạn chế / độ ưu tiên thấp |
|-----------------|---------------------------|
| Luồng nghiệp vụ quan trọng (đăng nhập, đặt hàng, thanh toán sandbox) | TC mới chưa chạy tay lần nào |
| TC lặp lại mỗi build | Yêu cầu thay đổi liên tục |
| TC nhiều bộ dữ liệu | Khám phá ad-hoc |
| TC dễ máy hóa (API rõ ràng) | TC phụ thuộc UI phức tạp, ít ổn định |

Tham chiếu thêm **Guideline Automation Testing** của FSOFT nếu đồ án yêu cầu chi tiết công cụ và nhóm RPA.
